# ERC-1400 아키텍처 상세 분석 (포괄적 보안 토큰 표준)

ERC-1400은 단일 스마트 컨트랙트가 아니라 여러 개의 작은 STO 표준들이 결합된 **엄브렐러(우산) 표준, 코어-플러그인 아키텍처**입니다. 

이를 통해 토큰 발행자(Issuer), 투자자(Investor), 그리고 규제기관(Regulator) 모두가 신뢰할 수 있는 단일 플랫폼을 구축합니다.

---

## 1. 뼈대: 왜 모음집(Umbrella) 일까?

법적인 실물 주식시장의 요구사항(문서 보관, 종류 주식 분리, 관리자 개입, 이체 통제 등)은 너무 방대합니다. 
그래서 이더리움 코어 개발자와 Polymath 팀은 이를 4개의 개별 표준으로 잘게 쪼개어, 각자의 역할만을 수행하게 한 뒤 합쳤습니다.

### 네 가지 핵심 기둥(Sub-standards)
1. **ERC-1410 (Partitions):** 토큰의 잔고를 성격에 따라 여러 파티션(트랜치)으로 쪼갭니다.
2. **ERC-1594 (Core Security Token):** 기본적으로 증권은 "허가된 자만 이체할 수 있다"는 제약사항 기능(`canTransfer`)을 제공합니다.
3. **ERC-1643 (Document Management):** 계약서 해시, 이사회 승인서, 백서 등을 블록체인 상에 영원히 저장하고 열람할 수 있게 만듭니다.
4. **ERC-1644 (Controller Operations):** 지갑 키 분실 및 압류 등에 대비하여 오퍼레이터(한화 등) 주소에서 토큰을 강제 회수/전송하게 만듭니다.

---

## 2. 보안 관리 주체 (Roles)

ERC-1400 내에서는 크게 세 종류의 행위자가 명확히 분리됩니다.

*   **Token Holder (지갑 소유자):** 실제 투자자. 자신의 한도 내에서 파티션을 지정하여 토큰을 송금할 수 있습니다.
*   **Issuer (발행사):** 새로운 토큰 파티션을 만들고 시장에 `issue` (발행)하는 권한.
*   **Controller / Operator (운영 관리자):** 홀더의 동의 없이도 법에 의해 토큰을 조정할 수 있는 전권(오프체인 규제의 강제성 반영). 반드시 실행 시 `operatorData`에 법적 근거를 남겨야 합니다.

## 3. ERC-1400 아키텍처 다이어그램 (Mermaid)

이 복합 표준이 어떻게 유기적으로 동작하는지 사용자(Actor) 측면과 내부 컨트랙트 모듈(Interfaces) 측면에서 분석한 다이어그램입니다.

```mermaid
graph TD
    classDef user fill:#6246ea,stroke:#333,stroke-width:2px,color:#fff;
    classDef standard fill:#e45858,stroke:#333,stroke-width:2px,color:#fff;
    classDef hub fill:#eebb4d,stroke:#333,stroke-width:2px,color:#000;
    classDef offchain fill:#2cb67d,stroke:#333,stroke-width:2px,color:#fff;

    INV(("👨‍💼 Investor <br>(Token Holder)")):::user 
    OPE(("👮‍♂️ Controller <br>(Operator/Issuer)")):::user
    
    subgraph "ERC-1400 Security Token Umbrella"
        1400{"ERC-1400 <br> Main Contract"}:::hub

        subgraph "Internal Sub-Standards"
            1400 --- P["📜 ERC-1410 <br> Partition Management"]:::standard
            1400 --- C["🔒 ERC-1594 <br> Core Security Checks"]:::standard
            1400 --- D["📁 ERC-1643 <br> Document Metadata"]:::standard
            1400 --- R["☢️ ERC-1644 <br> Recovery / Force Trasnfer"]:::standard
        end
    end
    
    DOC[("Off-chain Server <br> (IPFS/AWS)")]:::offchain 

    INV -->|"1. transferByPartition()"| 1400
    1400 -.->|"2. check canTransfer()"| C
    1400 -.->|"3. update balances"| P

    OPE -->|"A. forceTransfer() / redeem()"| 1400
    1400 -.->|"B. requires Operator Privilege"| R

    OPE -->|"C. setDocument() (Hash)"| 1400
    1400 -.->|"D. Store Document URI/Hash"| D
    D <...|"E. Link to PDF/Legal doc"| DOC
```

### 동작 과정 설명
1.  **일반 이체 절차:** 일반 투자자가 특정 파티션 토큰 송금을 명령하면, 1400본체는 **ERC-1594**를 호출해 권한 및 제약 사항을 검사하고, 이를 통과하면 **ERC-1410**의 파티션 잔고를 업데이트합니다.
2.  **강제 회수 절차:** 관리자가 법원 명령서를 등에 업고 강제 전송 명령을 내리면, 권한 검증은 **ERC-1644**가 담당하고 마찬가지로 잔고 업데이트를 진행합니다.
3.  **문서 증빙 절차:** 실제 한화와 투자자 사이에 작성된 약관이나 투자설명서 PDF 해시값이 **ERC-1643** 공간에 저장되어 위변조를 막습니다.

## 4. 모듈러 컴플라이언스(ERC-3643)와의 구조적 차이

*   **ERC-3643**은 법무 규칙을 담당하는 규정을 별도의 컨트랙트로 쪼갠 후(Pluggable), 메인 토큰은 가만히 놔두고 빈 껍데기만으로 검사를 외주(Delegate) 맡기는 구조입니다. (스마트 컨트랙트의 유지보수 특화)
*   **ERC-1400**은 "이 토큰은 태생부터 보통주, 우선주로 분리되어 있는 상품이고, 태생부터 강제 회수 권한을 제공해야 한다"라는 사상 하에 거대한 하나의 만능 칼(Swiss Army Knife)을 만들어두는 구조입니다. 모든 기능이 응집되어 있으나 다소 무거울(Heavy) 수 있습니다. 

어떤 실물 자산 상품을 다루느냐에 따라 한화 내부에서 어떤 표준을 선택할 것인지 건축 설계가 달라져야 합니다.
